#### Day 1（3/29 六）：後端框架觀念 + Shell Script
> 目標：搞懂 FastAPI/Node.js 在幹嘛 + 能寫 Shell 自動化腳本

**後端框架觀念釐清（3 小時）**
- [ ] 搞懂「為什麼需要後端框架」：
  - 你的記帳 app 如果要讓別人透過網路用 → 需要 API server
  - 框架幫你處理：路由、請求解析、回應格式、錯誤處理
- [ ] **FastAPI**（Python）：
  - 定位：Python 的現代 Web API 框架，自動產生 API 文件（Swagger UI）
  - 核心概念：路由裝飾器 `@app.get("/")` → 函數 → 回傳 JSON
  - 為什麼 DevOps 要懂：寫內部工具 API、健康檢查端點、webhook 接收器
  - 動手：把你的記帳 app 的功能用 FastAPI 包成 API（30 分鐘就能跑起來）
- [ ] **Node.js**（JavaScript）：
  - 定位：用 JavaScript 跑後端，非同步 I/O 高效能
  - Express.js 是最常見的框架（類似 Python 的 Flask）
  - 為什麼 DevOps 要懂：很多前端團隊用 Node.js，你要能幫他們建 CI/CD 和容器化
  - 不需要會寫，但要知道：`npm install`, `package.json`, `node_modules/`
- [ ] **Flask vs FastAPI vs Express 的差異**：
  | | Flask | FastAPI | Express (Node.js) |
  |--|-------|---------|-------------------|
  | 語言 | Python | Python | JavaScript |
  | 速度 | 較慢 | 快（async） | 快（event loop） |
  | API 文件 | 手動 | 自動產生 | 手動 |
  | 適合場景 | 小專案 | API 服務 | 前後端統一語言 |
- [ ] **RESTful API 核心觀念**：
  - HTTP 動詞對應操作：GET=讀、POST=建、PUT=改、DELETE=刪
  - 狀態碼：200 成功、201 已建立、400 請求錯誤、404 找不到、500 伺服器錯誤
  - JSON 是標準資料格式
  - 練習：用 `curl` 對你跑起來的 FastAPI 送 GET/POST 請求

**Shell Script（3 小時）**
- [ ] 變數、`if/else`、`for` 迴圈
- [ ] `$?` exit code、`$(command)` 命令替換
- [ ] 管線 `|` 與重導向 `>`, `>>`
- [ ] `set -euo pipefail`（安全模式，面試加分）
- [ ] 練習：寫一個系統健康檢查腳本（CPU、記憶體、磁碟使用率）

#### Day 8（4/5 六）：AI Agent 架構 + MCP 深入 + Token 優化
> 目標：掌握前沿 AI 架構，成為面試中的差異化武器

**AI 四代演進（必須能講出來）（1.5 小時）**
- [ ] 理解並能口述 AI 在 DevOps 的四代演進：
  - 第一代 Chatbot（2022-23）：純對話，不能執行
  - 第二代 Tool-Use（2023-24）：LLM 呼叫預定義函式
  - 第三代 Agentic（2024-25）：多步驟自主決策 + MCP 標準化
  - 第四代 Multi-Agent（2025-26）：多 Agent 協作 + 人在迴路
- [ ] 準備類比：「MCP 就是 AI 工具整合的 USB-C」

**MCP 深入（2 小時）**
- [ ] MCP 架構：Client（AI）↔ Protocol（JSON-RPC）↔ Server（工具）
- [ ] MCP 三大能力：Tools（函式呼叫）、Resources（資料讀取）、Prompts（範本）
- [ ] 動手：列出你的 Claude Code 目前用了哪些 MCP 功能
  - `claude mcp list` 查看已註冊的 MCP server
- [ ] 了解官方 MCP server 生態：GitHub、Filesystem、PostgreSQL、Slack 等
- [ ] 理解為什麼 MCP 比傳統 API 整合好：
  - 寫一次 server，任何 AI client 都能用
  - 安全性：server 跑在你的環境，資料不外流

**Token 優化五大策略（面試亮點）（1.5 小時）**
- [ ] **策略 1 — Tool Offloading**：讓 MCP tool 在 server 端處理資料，只回傳摘要
  - ❌ 把 5000 行日誌塞進 context → 貴又爛
  - ✅ MCP tool `log_search(pattern="500", last="1h")` → 回傳 3 行摘要
  - 類比：「這就像 DB 的 query pushdown，在資料源端就過濾」
- [ ] **策略 2 — Skills 按需載入**：20 個 skill 檔案只載入相關的 2-3 個
  - 你的實際體驗：Claude Code 的 Skills 機制
  - 節省 80-90% 的 context token
- [ ] **策略 3 — 子 Agent 分流**：spawn 子 Agent 處理獨立任務
  - 子 Agent 有自己的 context，不佔主 context
  - 只回傳結果摘要
- [ ] **策略 4 — 結構化輸出**：工具回傳 JSON 摘要而非原始資料
- [ ] **策略 5 — 工具定義精簡**：精簡 MCP tool schema，按需註冊

**Prompt Engineering 快速複習（1 小時）**

**低代碼 vs Agent-based 對比分析（2 小時）**
- [ ] 整理對比表：

  | 維度 | 低代碼 (n8n/Dify) | Agent (Claude Code + MCP) |
  |------|-------------------|--------------------------|
  | 流程彈性 | 固定 DAG，分支預先定義 | 動態決策，根據情境調整 |
  | 錯誤處理 | 預設 retry/fallback | AI 理解原因，智慧應變 |
  | Long tail | 每種情況都要手動加規則 | AI 處理從未預見的場景 |
  | 維護成本 | workflow 越多越難維護 | Skills/MCP 模組化 |
  | 適合場景 | 每天 9AM 發報告 | Incident response、除錯 |

- [ ] 準備具體例子：
  ```

  Agent：讀日誌 → 理解原因 → 判斷嚴重性 → 自動採取最適當行動
  ```

- [ ] **論述 1**：「AI 在 DevOps 不只是 Chatbot」

- [ ] **論述 2**：「Token 優化不只是省錢，更是品質問題」

  > 正確做法是 Tool Offloading — 像 DB 的 query pushdown 一樣，在資料源端過濾。

- [ ] **論述 3**：「低代碼和 Agent 是互補的」

  > 最佳架構是混合使用，中間用 MCP 連接。
  > 不是要取代 n8n，而是讓 AI 處理 n8n 處理不了的 long tail。

- [ ] **論述 4**：「Skills 機制可以將團隊知識程式碼化」
  > 把部署檢查清單、incident response 流程、code review 標準寫成 Skills。

- [ ] **論述 5**：「我不只是理論，我是實際使用者」

  > 我的學習筆記用 Git 管理、用 AI 輔助整理，這就是 AI-native 工作流。
