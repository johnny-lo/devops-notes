# DevOps 接手新專案的思考框架

> 日期：2026-03-27
>

---

## 重點整理

- DevOps 工程師的工作 = 觀察整條交付鏈，找到手動、脆弱、緩慢的環節，用自動化和標準化逐步替換
- 接手專案不只看系統，更要搞清楚環境、資料、權限
- 監控先於自動部署 — 看不見問題就不要動生產環境
- 不要想一次全做，按層次逐步建設
- 最重要的思維習慣：持續問「如果這裡壞了會怎樣」

---

## 接手節奏

先有全局感再動手，建議的 milestone：

| 時間 | 目標 | 對應 Step |
|------|------|-----------|
| Day 1 | 搞懂系統做什麼、取得必要權限 | Step 1 |
| Week 1 | 環境/權限/資料盤點、畫架構圖、識別痛點 | Step 2、3 |
| Month 1 | 交付鏈全面盤點、top 3 痛點、基本監控、第一個 quick win | Step 4、5、6 |

這是參考時間，實際會依專案大小和複雜度調整。重點不是卡時間，而是確保有節奏地推進，不會在某一步卡太久。

---

## Step 1：搞懂「這是什麼」

還沒碰任何設定之前，先回答：

- 這個系統**做什麼**的？給誰用？
- 用什麼**技術棧**？（語言、框架、資料庫）
- **架構**長什麼樣？（單體？微服務？有幾個元件？）
- 有沒有**文件**？（README、wiki、架構圖）

> **驗證方式：** 能向別人口頭解釋這個系統做什麼，畫出架構圖讓人 review。如果解釋不清楚，代表還沒搞懂。

---

## Step 2：盤點環境、權限、資料

在動手改任何東西之前，先搞清楚你面對的是什麼環境、誰有什麼權限、資料怎麼管。這一步漏掉的東西，後面都會變成地雷。

### 環境盤點

- 有幾個環境？（dev / staging / production）
- 環境之間的差異是什麼？（這是踩雷重災區 — config 不同、版本不同、資料量不同）
- 網路拓撲長什麼樣？防火牆規則？DNS 怎麼設的？

### 權限與存取

- 誰有 production access？存取方式是什麼？（SSH、VPN、console）
- Secrets / credentials 怎麼管理？存在哪？（vault、env var、config file）
- SSH key / API token 有沒有定期 rotate？
- 離職人員的權限有沒有收回？

### 資料

- 資料庫備份策略是什麼？備份有驗證過能成功還原嗎？
- 有 migration 機制嗎？（schema 變更怎麼做）
- 資料保留政策？有合規要求嗎？（GDPR 等）

> **驗證方式：** 能列出所有環境、存取方式、備份策略，不靠別人提示。畫一張環境拓撲圖。

---

## Step 3：摸清「現在怎麼跑」

沿著交付鏈逐一確認：

```
程式碼 → 怎麼 build → 怎麼測試 → 怎麼部署 → 跑在哪裡 → 怎麼監控
```

| 環節 | 要問的問題 |
|------|-----------|
| 程式碼 | repo 在哪？branching 策略？有 code review 嗎？ |
| Build | 怎麼打包？有自動化嗎？還是手動？ |
| 測試 | 有 unit test 嗎？涵蓋率多少？有跑 lint/security scan 嗎？ |
| 部署 | 怎麼交付給使用者？手動複製？自動部署？ |
| 運行環境 | 跑在哪？本機？VM？容器？雲端？ |
| 監控 | 壞了怎麼知道？有 log 嗎？有告警嗎？ |
| 資料 | 資料怎麼流動？有 ETL 嗎？有排程 job 嗎？ |
| 環境差異 | dev 和 production 的差異有文件化嗎？有沒有「只在 prod 出現」的設定？ |

每一格如果答案是「沒有」或「手動」，就是可以改善的點。

> **驗證方式：** 表格每一格都填滿，沒有空白或「不確定」。如果有不確定的，就是還需要繼續挖的地方。

---

## Step 4：找到「最痛的地方」

排優先順序不能只看頻率和痛苦，要用四個維度評估：

| 維度 | 問自己 |
|------|--------|
| 頻率 | 這個問題多常發生？ |
| 痛苦程度 | 每次發生有多痛？（時間、風險、情緒成本） |
| 影響範圍 | 只影響一個人，還是影響整個團隊甚至使用者？ |
| 實作成本 | 修好它要花多久？一天還是三個月？ |

原則：

1. **高頻 × 高痛苦 × 廣影響 = 先做**
2. **Quick Wins 先行** — 低成本但中等效益的事情優先處理，快速建立信任和動能。一個中等痛但一天能搞定的改善，往往比一個超痛但要三個月的大工程更值得先做
3. **什麼東西出錯最嚴重？** → 優先加防護網

> **驗證方式：** 列出 top 3 痛點，每個都能說清楚為什麼排這個順序（用四個維度解釋）。

---

## Step 5：按層次逐步建設

建議的建設順序（監控在自動部署之前，因為看不見問題就不該自動推東西出去）：

```
Level 0: 版本控制（Git）
    ↓
Level 1: 監控與可觀測性（先看得見，才改得動）
    ↓
Level 2: 自動化測試（CI）
    ↓
Level 3: 自動化建置（CD - build）
    ↓
Level 4: 基礎設施即程式碼（IaC — 環境可重建性）
    ↓
Level 5: 自動化部署（CD - deploy）
```

**注意：這是建議順序，不是硬性規定。** 現實中很多專案是先有 CD 再補測試的，這個框架的目的不是叫你照順序走，而是幫你意識到「跳過某一層的風險是什麼」。根據專案現狀可以調整，但要有意識地做選擇，而不是因為不知道所以跳過。

> **驗證方式：** 能標出專案目前在哪一層，能說出「為什麼決定先做這層而不是那層」。

## Step 6：持續問「如果這裡壞了會怎樣」

| 如果... | 對應的防護機制 |
|---------|--------------|
| CI server 掛了 | 團隊能不能手動 build？有備案嗎？ |
| 部署失敗 | 能不能 rollback？要多久？ |
| 半夜 production 掛了 | 誰會收到通知？怎麼查問題？ |
| 有人把 secret 推上 repo | 有沒有 pre-commit hook 或掃描機制？ |
| 依賴套件有漏洞 | 有沒有定期掃描？（如 pip-audit） |
| 資料庫掛了 / 資料損毀 | 備份能還原嗎？RPO / RTO 是多少？ |
| 流量暴增 | 有 auto-scaling 嗎？有 rate limiting 嗎？ |
| 上游 API / 第三方服務掛了 | 有 fallback 機制嗎？有 circuit breaker 嗎？ |
| DNS 設定錯誤 | 有監控 DNS 解析嗎？TTL 設多少？ |
| Certificate 過期 | 有自動 renew 嗎？有到期告警嗎？ |

每個「如果」對應一個你可以建設的防護機制。

**光列出來不夠，要實際演練。** 寫好 rollback 步驟跟真的能 rollback 是兩回事。建議：

- 定期做 runbook 演練（至少對最關鍵的故障場景）
- 在非 production 環境實際測試 rollback 流程
- 考慮引入 chaos testing（在可控範圍內製造故障，驗證防護機制是否真的有效）

> **驗證方式：** 至少對 top 3 故障場景實際演練過一次，確認防護機制真的能動。

## 不同場景的調整提示

這個框架以單體應用為起點，但不同場景需要調整：

- **微服務 / 多 repo：** Step 3 的交付鏈表格需要每個服務各一份，額外關注服務間通訊（API contract、message queue）和跨服務部署順序
- **容器化 / K8s：** Step 2 的環境盤點需包含 cluster 配置、image registry、namespace 規劃；IaC 的優先級可能需要提前
- **多人協作：** 環境和權限盤點變成最關鍵的一步，溝通成本也要納入 Step 4 的評估維度

## 實際案例：Automatic-beadblaster

| 環節 | 接手前 | 現在 |
|------|--------|------|
| 程式碼 | GitLab repo | 同 |
| Build | 手動跑 PyInstaller | CI 自動打包（[[gitlab_ci_pipeline_guide]]) |
| 測試 | 無 | 112 個 unit test + ruff lint |
| 安全 | 無 | pip-audit 掃描依賴漏洞 |
| 部署 | 手動從 dist/ 複製 | 從 artifacts 下載（仍為半手動） |
| 監控 | 無 | 無 |

目前在 **Level 3**（自動化建置）。下一步方向：Level 1 監控與可觀測性（補上目前的盲區）。

---

## 踩過的坑

- 想一次做太多 → 應該從最痛的點開始，一次解決一個問題
- 沒有測試就自動部署 → 等於自動把 bug 推出去，更危險
